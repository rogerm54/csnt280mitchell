Objective 5.2 - Using scripts to normalize data for insertion into relational database tables

Suppose your organization pays a company to maintain the organization data in some cloud-based application. If your organization chooses to change the vendor, the old vendor will not be likely to export all your data in nice tables. Doing so would be time consuming, but it would also give too clear an idea of how their database was set up. It is far more likely that they will give the data back from an exported view.

Since a view combines several tables, the data will not be normalized. What we will look is an example of such data, and look at how we might use scripts to normalize the data. Here is a file, "signups_view.txt" that contains data for a system that allowed students to sign up for courses:

As you can see, this is just a bunch of data showing students that signed up for courses. The data is clearly not normalized. If this was your organization's data, someone in your organization should have a good idea of what this data means. Working with such a person would be helpful in getting this data normalized.

The titles will help you to separate out the data. You would also have to make some guesses about the id columns. The cid column seems to be a course id, as the same course has the same value for a given cid. The st_id column is probably a student id as the same student has the same value for st_id. The last id column, su_id has no repeats. It is most likely an id associated with the signups for a course.

When you separate out the data, you need to keep in mind how the data in a given row is related. For example, Jane Doe signed up for CSNT231, and we need to keep this relationship. This suggests that we need a bridge table to join the courses and students. So, let's try to sort things out by assuming that we will have a courses table, a students table and a signups table to join the courses table with the students table.

When you have a join (bridge) table, you have to maintain the relationship in a given row of data. If you are using Python, this means that you are dealing with dictionaries on each row, that will be stored in lists. If you are using JavaScript, the comparable structure would be having JSON objects on each row, being stored in arrays. Let's look at some scripts in each language that will help to separate out the data.

infile = open("signups_view.txt","r")

This program is quite simple. The file is opened on line 2 and the first line is read in on line 3. Since we don't need the header line, we read it in but don't do anything with it. const fs = require('fs');
fs.readFile("signups_view.txt", function(err, buff) {
   let data = [];
   data = buff.toString().split("\n");
   course_list = [];
   course_ids = [];
   student_list = [];
   student_ids = [];
   signup_list = [];
   for (let i = 1; i < data.length - 2; i++) {
      let tokens = data[i].trim().split("|");
      let course = {};
      let student = {};
      let signup = {};
      course.id = Number(tokens[0]);
      course.prefix = tokens[1];
      course.number = tokens[2];

The code is very similar to the Python code. There are some noticeable differences on line 10, lines 36-38 and lines 39-41. On line 10, note that the for loop must be stopped before data.length -2. This is because the way that JavaScript reads in the file, an extra line is read in. Lines 36-38 are the way that an array of JSON objects is sorted. This is similar to the way that Python2 used to sort lists of dictionaries. But, with Python3, there is a fancier way using keys. Lines 39-41 sort the student_list array by id. Again, this is different from the code for the Python program that sorts the comparable list.

The way that JavaScript deals with JSON objects is a little simpler than Python dictionaries. If you have a dictionary named course and you want to get the id key, the Python code looks like this: 


course["id"]

the correspond JavaScript code would be: 


course.id

You can use the same notation as the Python notation in JavaScript, but it is simpler to use the dot notation (as long as the key name does not contain any spaces).
