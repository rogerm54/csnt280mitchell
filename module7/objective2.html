Objective 7.2 Querying a MongoDB database

Assuming the database we built on the last page . Let's do some queries:

Querying a single collection

We have already done some simple queries of single collections. Let's go over that in a little more detail. Just in case you have stopped the Docker container, mongo_cont , here are the steps to start that container and get back inside the registrations database: 


$ cd ~/Desktop/node_area/docker_mongo/docs/registrations
$ docker start mongo_cont
$ docker exec -it mongo_cont bash
# cd registrations
# mongo
> use registrations

Here are some simple queries to show all the documents in the three collections. The first query is to find all the documents in the students collection: 


> db.students.find()

Querying multiple related collections

Finally, let's see how we can run a query that joins the students , sections and student_section collections: 


> db.student_section.aggregate([
... { $lookup: { from: "students", localField: "student_id", foreignField: "_id", as: "Student"}},
... { $lookup: { from: "sections", localField: "section_id", foreignField: "_id", as: "Section"}},
... { $project: {"Student.first_name":true,"Student.last_name":true,"Section.title": true}}
... ])
{ "_id" : 1, "Student" : [ { "first_name" : "Jane", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT110" } ] }
{ "_id" : 2, "Student" : [ { "first_name" : "Jane", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT132" } ] }
{ "_id" : 3, "Student" : [ { "first_name" : "John", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT132" } ] }
{ "_id" : 4, "Student" : [ { "first_name" : "John", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT140" } ] }
{ "_id" : 5, "Student" : [ { "first_name" : "Bill", "last_name" : "Gates" } ], "Section" : [ { "title" : "CSNT110" } ] }
>

To try this out, it is probably best to just copy lines 1-5 and use SHIFT-CTRL-V to paste them into the Mongo environment. That way, you will get the command in without any typographical errors. Line 1 starts of the aggregate() function call on the student_section collection. The argument to the aggregate() function is an array that holds several JSON objects. In this case, line 2 specifies a JSON object that consists of a call the the $lookup function. Here is that call spread out over several lines so that you can see the parts more clearly: 


$lookup: {
   from: "students",
   localField: "student_id",
   foreignField: "_id",
   as: "Student"
}

So, this is calling for a query from the "students" collection where the "student_id" key from a "student_section" object matches the "_id" key from the "students" collection. The object obtained from this match is given the alias, "Student".

So, looking back at line 3, you can see that this line is returning an object that is the result of querying the "sections" collection with the "section_id" key of a "student_section" object matching the "_id" key of the "sections" collection. The object returned by this match is given the alias, "Section".

Finally, line3 uses a call to the $project function. Here is that call spread out over several lines: 


$project: {
  "Student.first_name": true,
  "Student.last_name": true,
  "Section.title": true,
}

This is just a function that will make it so that the key:value pairs returned will be the "first_name" from the "Student" alias, the "last_name" from the "Student" alias, and the "title" from the "Section" alias.

You can see that these are the key:value pairs returned on lines 6-10. Lines 6-10 show the objects that is similar to what is displayed in a view that joins tables in a relational database.

Here are two more examples of using aggregation and $lookup. The first example is used to return the objects that represent the students taking CSNT132 : 


> db.student_section.aggregate([
... { $lookup: { from: "students", localField: "student_id", foreignField: "_id", as: "Student"}},
... { $lookup: { from: "sections", localField: "section_id", foreignField: "_id", as: "Section"}},
... { $project: {"Student.first_name":true,"Student.last_name":true,"Section.title": true}},
... { $match: {"Section.title": "CSNT132"}}
... ])
{ "_id" : 2, "Student" : [ { "first_name" : "Jane", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT132" } ] }
{ "_id" : 3, "Student" : [ { "first_name" : "John", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT132" } ] }

Lines 1-4 are mostly the same as the other lookup. The only difference is the comma at the end of line 4 that is needed because of the additional condition on line 5. Line 5 is similar to a where clause, in that if filters the results so that only those with the "Section.title" key matching "CNST132" will be returned.

The second example is used to return the objects that represent the sections signed up for by Jane Doe: 


> db.student_section.aggregate([
... { $lookup: { from: "students", localField: "student_id", foreignField: "_id", as: "Student"}},
... { $lookup: { from: "sections", localField: "section_id", foreignField: "_id", as: "Section"}},
... { $project: {"Student.first_name":true,"Student.last_name":true,"Student._id":true,"Section.title": true}},
... { $match: {"Student._id": 1}}
... ])
{ "_id" : 1, "Student" : [ { "_id" : 1, "first_name" : "Jane", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT110" } ] }
{ "_id" : 2, "Student" : [ { "_id" : 1, "first_name" : "Jane", "last_name" : "Doe" } ], "Section" : [ { "title" : "CSNT132" } ] }

Line 4 has been modified so that "Student._id" is included in the projected objects. Line 5 just filters those objects so that the objects returned are for "Student._id" equaling 1 (which is Jane Doe's _id value).

Querying before $lookup existed

Before the $lookup was added, there was nothing to join collections. In fact, many NoSQL databases don't even have the concept of collections. So, to store the data that we have, you would have to store everything about a student in one object. For Jane Doe, this would be an example of how such an object might be stored: 


{
   _id: 1,
   first_name: "Jane",
   last_name: "Doe",
   sections: [
      {
         _id: 100,
         title: "CSNT110",
         credits: 3
      },
      {
        _id: 101,
        title: "CSNT132",
        credits: 4
      }
   ]
}

For each "student" document like this, you would have repeat information for any sections that are signed up for by more than one student. The data is not normalized. Instead, the data is stored in a hierarchical fashion like this. As you can imagine, having to query many "student" documents like this to find the students in the CSNT132 course would be tricky for people coming from the relational database world.

This is why some people say that relational data should not be store in NoSQL databases. This has been one of the problems with the Electronic Health Records (EHR) requirement for medical insurance that comes from the Affordable Care Act. Many hospitals and physicians uses systems like Epic (https://www.epic.com) to store patient records. Epic stores the patient records in a hierarchical fashion in a NoSQL database system called MUMPS. One of the criticisms of Epic was raised in How Google EHR Could Fix One of Healthcare's Epic Flaws is that it has a poor user interface.

Besides having a poor user interface, storing the data in a hierarchical fashion where all the documents are based on single patients, makes it hard to perform queries that need to get at multiple patients at once. For a given patient, it is easy to find anything stored in that patient's document providing you know the key to lookup the value. But, if you want to use data like this to find out something like what percent of male Asians in Hawaii have Type II diabetes, this is harder to do. If that same data were reorganized into a normalized relational database, that query would be easy to do.

✴ ✴ ✴ ✴ ✴

This is another reason why it is important for more people to become familiar with NoSQL databases so that either they don't use NoSQL databases to store relational data or they figure out good ways to store the data in a NoSQL database while providing the ability to perform the type of queries needed.

✴ ✴ ✴ ✴ ✴

When the major purpose of a database requires accessing hierarchical data, then a NoSQL database makes sense and will scale better than a relational database. However, when you need to perform horizontal queries of data that is stored in a hierarchical fashion, this will involve recursive processes that often are much slower than if the data were stored in a relational database. At the same time, storing data in NoSQL databases allow the kind of scaling that will lend itself well to machine learning processes. So, the challenge of deciding between NoSQL and relational databases is very much a hot topic at this point in time.
