Docker containers with databases used with Node.js

We have already used docker-compose to create and connect two docker containers so that we can use PHP and Apache with a PostgreSQL database. If we decide to use Node.js with a database, we can use just a single docker container for that database. Let's look at how we can set something like this up.

Creating a Node.js package area

We can start by creating a directory that we can use to install all the npm packages needed for working with different databases. Eventually, we want to be able to work with a PostgreSQL database and also a MongoDB database. We will do this by creating the following directory ~/Desktop/node_area . Then, we will initialize this directory so that we can install the various npm packages needed to interact with the databases. Run the following commands: 


$ mkdir ~/Desktop/node_area
$ cd ~/Desktop/node_area
$ npm init -y

The last command will create a file "package.json" that will be used to keep track of the installed npm packages. Next, we can install the following packages: express , pg , mongo and cors . Within the ~/Desktop/node_area directory, run the following command: 


$ cd ~/Desktop/node_area
$ npm install express pg mongoose cors

The express package is used to help set up the routes that are used for a REST-like API between a backend database and a web interface. The pg package is used to connect to the PostgreSQL database from a Node.js script. The cors package would be needed to test and create a web interface for the database. the mongoose package is used to connect to a MongoDB database from a Node.js script.

It is important to understand that Node.js scripts are JavaScript scripts that are meant to run on the server side. So, we will be creating such scripts to interact with the databases from the command line. But, the express package will make it so that we expose those database connections to client-side JavaScript that is run inside a browser. However, well before we work on the web interface, the backend must be developed and tested.

In a similar manner, if we are using PHP/Apache for a web interface, we still need to set up the backend database before we deal with any of the web interface.

Setting up a Docker container that contains PostgreSQL

Let's setup a Docker container that will be used to provide a PostgreSQL server. This will require a fairly simple Dockerfile for the image. Once the image is built, we will create a container for that image that will allow us to use PostgreSQL. Let's start with the following commands: 


$ cd ~/Desktop/node_area
$ mkdir docker_pg
$ cd docker_pg
$ mkdir docs

Within the ~/Desktop/node_area/docker_pg folder, we want to create a file called "Dockerfile": 


$ cd ~/Desktop/node_area/docker_pg
$ geany Dockerfile &

Here are the contents of "Dockerfile": 


FROM postgres:14-alpine
RUN apk add --no-cache nano
WORKDIR /docs

In that same directory, create the script "create_image.sh": 


$ cd ~/Desktop/node_area/docker_pg
$ geany create_image.sh &

Here is the contents of "create_image.sh": 


docker build -t pg_img .

Set the permissions on that file so that it can be executed. Then run the script to create the image called "pg_img": 


$ cd ~/Desktop/node_area/docker_pg
$ chmod u+x create_image.sh
$ ./create_image.sh

This will create a docker image called pg_img that has that will run the PostgreSQL service and also have the nano text editor available. Next, let's create a script called "create_container.sh" that will be used to create the Docker container that will run the image we just created. Here is the contents of "create_container.sh": 


docker container run -it --name=pg_cont --volume $(pwd)/docs:/docs \
--env POSTGRES_PASSWORD='pgsql_man' pg_img

Use "chmod u+x create_container.sh" to make this executable and then run this shell script. This will create a docker container named pg_cont (using the --name switch) from the pg_img docker image. This --volume switch will create a persistent volume that will map the docs directory in the current directory to a directory /docs in the container. The -it switch will run the container as an interactive terminal. Without the --env (environment variable switch), the container would start and exit immediately. But, starting this way, we can hit CTRL-C to stop the container, and the container gets created in a way that will allow the container to remain open. This is because the maintainers at Docker Hub must have put in something where failing to set a password for the postgres user is not desirable.

We need to start up the container, pg_cont again, and then make some changes. 


$ docker start pg_cont

when the container starts up, run the following command: 


$ docker exec -it pg_cont bash

This will start up an interactive root terminal inside the pg_cont container. From there we can do the following: 


# su postgres
# psql postgres
# create role csnt280man login createdb createrole superuser;
# alter role csnt280man encrypted password 'pgsql_man';
# create database csnt280db owner csnt280man;
# exit
# exit

The first line switches to the postgres user. The second line starts up psql and connects to the postgres database. The third line creates the csnt280man super user. The fourth line sets 'pgsql_man' to be the password for the csnt280man user. The fifth line creates a database called csnt280db and makes csnt280man the owner of that database. The sixth line exits from the switch into the postgres user. The last line causes an exit from psql.

✴ ✴ ✴ ✴ ✴

The shell scripts, "create_image.sh" and "create_container.sh" are just one line script that could have just been typed at the console. But, by putting the commands into those scripts, we can always view those scripts to see how the docker image and docker container have been created. Knowing how the image and container have been created is important if you have not run the container for awhile and you need to remember how the container and the image the container is based upon were originally created.

To make the PostgreSQL service more secure, we want to make a modification to the file /var/lib/postgresql/data/pg_hba.conf.

Here are the steps to follow. Before executing these steps, make sure that the pg_cont docker container is running. If it is not running, enter "docker start pg_cont" at the command line to start it up. 


$ cd ~/Desktop/node_area/docker_pg
$ docker exec -it pg_cont bash
# nano /var/lib/postgresql/data/pg_hba.conf

Once you are inside nano type ALT-G, followed by 84 (to go to line 84). That line currently looks like this: 


local     all     all     trust

Comment that line out and replace it with this line: 


local     all     all     md5

That is, you are replace "trust" with "md5". In nano type CTRL-O (output) and hit an Enter. Then, type CTRL-X (to exit nano). This means that a user will have to enter the password to use a PostgreSQL database. To make this take effect, you need to exit from the Docker container, stop the container and start the container again. If you did this correctly, you will need to use the 'pgsql_man' password to get into the csnt280db database. Here is a screen shot of an interactive session to show this.

The second time, I supplied the correct password so that I got into the csnt280db database as shown. Note that because csnt280man is a superuser, the prompt has a # at the end.

Now that we are inside the csnt280db database, let's create the regular user bob and give bob a database named 'bobdb': 


csnt280db=# create role bob login;
CREATE ROLE
csnt280db=# alter role bob encrypted password 'somepass';
ALTER ROLE
csnt280db=# grant bob to csnt280man;
GRANT ROLE
csnt280db=# create database bobdb owner bob;
CREATE DATABASE

To verify that this worked, exit the csnt280db database and login to the bobdb database as bob: 


csnt280db=# \q
root@dc7a8bfc4654:/docs# psql -U bob -W bobdb
Password:

Type "help" for help.

bobdb=>

Since 'bob' is a regular user, the prompt ends with a > (instead of a # like it does for csnt280man). We will make use of this container later. Let's create a container that is used to provide MongoDB databases.

A Docker container for MongoDB

We can start by going to a terminal and creating a directory to hold files for setting up a Docker container for MongoDB: 


$ docker stop pg_cont
pg_cont
$ cd ~/Desktop/node_area/
$ mkdir docker_mongo
$ cd docker_mongo/
$ geany Dockerfile &

First, we stop the pg_cont container. Then, change into the directory that it initialized for Node.js and make a directory called "docker_mongo". Change into that directory and use Geany to create Dockerfile with these contents: 


FROM mongo:4.2-bionic
WORKDIR /docs

Create a subdirectory called "docs" and make a script that can be used to create the MondoDB image: 


$ mkdir ~/Desktop/node_area/docker_mongo/docs
$ cd ~/Desktop/node_area/docker_mongo
$ geany create_image.sh

Here is the contents of "create_image.sh": 


docker build -t mongo_img .

Make "create_image.sh" executable and run it to create the mongo_img image: 


$ chmod u+x create_image.sh
$ ./create_image.sh

To make sure the container runs properly on the Apporto virtual desktops, use a name like this: 


mongo_cont_jd

for your container. This assumes that your initials are jd (for Jane Doe). So, substitute in your own initials to avoid a namespace collision.

So, create the "create_container.sh" file with these contents: 


docker container run -d --name=mongo_cont_jd -v $(pwd)/docs:/docs mongo_img

Make "create_container.sh" executable and run it to create the mongo_cont container: 


$ chmod u+x create_container.sh
$ ./create_container.sh

Stop the mongo_cont_jd container. Then, restart that container and get to a root prompt inside the mongo_cont container: 


$ docker stop mongo_cont_jd
$ docker start mongo_cont_jd
$ docker exec -it mongo_cont bash

Once inside the mongo_cont container, run mongo and create a database called "bobdb" and give this database some records:

After starting up mongo by typing "mongo" at the prompt, the MongoDB server starts up. You will see the MongoDB version and some messages. When you get a prompt, type the "show dbs" command to list all the databases. Then type the "use bobdb" command to create a database named bobdb Like other NoSQL databases, MongoDB saves everything as JSON objects. This line: 


db.students.save({first_name: "Jane", last_name: "Doe"});

will create a collection called "students" and store the JSON object with the first_name and last_name key: value pairs. As can be seen, there will now be three JSON objects stored in the "students" collection. A collection in MongoDB is similar to a table in a relational database.

To query that database from within the mongo prompt, you do this: 


> db.students.find({})
{ "_id" : ObjectId("5d99634427bba89d1b4bc224"), "first_name" : "Jane", "last_name" : "Doe" }
{ "_id" : ObjectId("5d99634b27bba89d1b4bc225"), "first_name" : "John", "last_name" : "Doe" }
{ "_id" : ObjectId("5d99635927bba89d1b4bc226"), "first_name" : "Bill", "last_name" : "Gates" }

By leaving the curly braces ({ }) empty on the first line, all the objects in the students collection are returned.
