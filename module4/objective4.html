Objective 4.4 - Deleting data using a SQL query

Dropping a table

Sometimes you need to drop an entire table. Usually, you do this by running your "init.sql" script as that drops all the tables before recreating the tables. But, if you just want to drop a single table without recreating it, the syntax is: 


drop table <tablename> [if exists] [cascade];

The optional, 


if exists

is useful if you are running a SQL script to drop the table. If the table does not exist, you will get a NOTICE, but you won't get an ERROR.

the optional clause, 


cascade

will cause any objects (tables, views, sequences) that depend on the table to be dropped as well.

Deleting records from a table

The basic syntax for deleting a record is this: 


delete from <tablename> where <condition>;

If you don't specify the where clause, all the records from that table will be deleted. Consider the "bookstore_db" database that was created earlier. Here is the "bookstore_init.sql" script that was used to initialize that database: 


-- bookstore_init.sql
drop table if exists books cascade;
drop table if exists customers cascade;
drop table if exists sales cascade;

create table books (
   id serial,
   title varchar(100),
   author varchar(100),
   price decimal(6,2),
   primary key (id)
);

create table customers (
   id serial,
   first_name varchar(30),
   last_name varchar(40),
   primary key (id)
);

create table sales (
   id serial,
   customer_id integer references customers(id),
   book_id integer references books(id),
   primary key (id)
);

create or replace view sales_view as
   select customers.id as cid, customers.first_name,
   customers.last_name, books.id as bk_id,
   books.title, books.author, books.price from
   customers join sales on customers.id=sales.customer_id
   join books on sales.book_id=books.id;

These tables were populated using the SQL script, "add_initial_data.sql" shown next: 


insert into customers (id, first_name,last_name) values
   (1, 'Jane','Doe');
insert into customers (id, first_name,last_name) values
   (2, 'John','Doe');
insert into customers (id, first_name,last_name) values
   (3, 'Bill','Gates');

insert into books (id, title,author,price) values
   (1, 'Go Dog Go','P.D. Eastman',7.99);
insert into books (id, title,author,price) values
   (2, 'The Cat in the Hat','Dr. Seuss',8.99);
insert into books (id, title,author,price) values
   (3, 'Ender''s Game','Orson Scott Card',9.99);

insert into sales (customer_id,book_id) values (1,1);
insert into sales (customer_id,book_id) values (1,2);
insert into sales (customer_id,book_id) values (2,1);
insert into sales (customer_id,book_id) values (2,3);
insert into sales (customer_id,book_id) values (3,1);
insert into sales (customer_id,book_id) values (3,2);
insert into sales (customer_id,book_id) values (3,3);

Suppose that Jane Doe decides that she wants to return Go Dog Go, we could do this by deleting the appropriate record from the sales table. Unfortunately, when we created the sales_view view, we did not include the id from the sales table. If we had, we could just look up the sales_id from the sales_view.

We could look up the id for the sale, but we can also just delete the record by using the correct customer id and book_id. Here is the delete query: 


delete from sales where customer_id=1 and book_id=1;

Here is the result of running that delete query: 


bookstore_db=> delete from sales where customer_id=1 and book_id=1;
DELETE 1
bookstore_db=> select * from sales_view;

As you can see in the select query for the sales_view view, the sale of Go Dog Go to Jane Doe is gone.

Deleting a row that other objects depend on

Suppose that Bill Gates made his purchases on a credit card and you found that the credit company refused to pay. So, you need to delete all of those sales. In fact, you are so angry at Bill Gates, that you decide to delete him as a customer. The delete query actually depends on the type of database you are using. For PostgreSQL, you would do something like this: 


delete from sales where customer_id=3;
delete from customers where id=3;

In other words, you have to manually delete records so in a way that does not violate any foreign key constraints. If you had just tried to delete the customer without deleting the sales records associated with that customer, you would get a foreign key violation and the delete query would fail. So, here is how you could dump Bill Gates: 


bookstore_db=> delete from sales where customer_id=3;
DELETE 3
bookstore_db=> delete from customers where id=3;
DELETE 1
bookstore_db=> select * from customers;
 id | first_name | last_name 
----+------------+-----------
  1 | Jane       | Doe
  2 | John       | Doe
(2 rows)

bookstore_db=> select * from sales_view;
 cid | first_name | last_name | bk_id |       title        |      author      | price 
-----+------------+-----------+-------+--------------------+------------------+-------
   1 | Jane       | Doe       |     2 | The Cat in the Hat | Dr. Seuss        | 8.99
   2 | John       | Doe       |     1 | Go Dog Go          | P.D. Eastman     | 7.99
   2 | John       | Doe       |     3 | Ender's Game       | Orson Scott Card | 9.99
(3 rows)

As you can see from querying the customers table, Bill Gates has been deleted. And, from the sales_view you can see all the sales associated with Bill Gates are also deleted.

Later on, we will discuss NoSQL databases. Deleting records in NoSQL databases will also require manually deleting a record's dependent objects. However, since there are no relations (foreign key constraints), the database software itself will not protect you from messing up the integrity of your data. It is up to you, to main the integrity by performing the corrects steps in the correct order.
