Objective 3 - Creating views that combine several tables in a relational database

In the previous lesson we wound up with the following SQL script, "init.sql": 


-- init.sql
drop table if exists students cascade;
drop table if exists sections cascade;
drop table if exists signups cascade;

create table students(
   id serial,
   first_name text,
   last_name text,
   major text,
   primary key (id)
);
create table sections(
   id serial,
   prefix text,
   number text,
   credits integer,
   primary key (id)
);
create table signups(
   id serial,
   student_id integer references students(id),
   section_id integer references sections(id),
   primary key (id)
);

Let's create a view that joins all three tables: 


create view signups_view as
   select students.id as st_id, students.first_name, students.last_name,
   students.major, sections.id as sect_id, sections.prefix,
   sections.number, sections.credits from 
   students join signups on students.id=signups.student_id
   join sections on signups.section_id=sections.id;

On line 1, we use the SQL CREATE VIEW AS to create a view called "signups_view". Line 2 starts of with select followed by all the fields from the tables that we want to include in the view. Note on line2 that we use the following alias, 


students.id as st_id

This is because we refer to a field with the same name later on with sections.id . To avoid a collision between having two fields with the same name, we use the alias st_id for the students.id field. This is why the other alias, 


sections.id as sect_id

is used as well. Although we might not use every field selected when using this view in an application, you should include any field that you might possibly want to display. The end of line 4 has the key word from that designates where those fields are going to be selected from. Lines 5 and 6 are the lines used to join the three tables together. Line 5 shows this: 


students join signups on students.id=signups.student_id

This comes directly from the foreign key constraint imposed by the definition of the signups . From the declaration of the signups table on line 22, we have the student_id field declared as: 


student_id integer references students(id)

As is shown in the view, this means that students.student_id must match students.id .

Similarly, line 23 from the definition of the signups table: 


section_id integer references sections(id)

means that signups.section_id must match sections.id .

Inserting some data

To see the view in action, let's insert some data into the three tables. Here is a SQL script called "add_initial_data.sql" that can be used for this purpose: 


-- add_initial_data.sql
insert into students (first_name, last_name, major) values 
   ('Jane','Doe','CSNT');
insert into students (first_name, last_name, major) values 
   ('John','Doe','CSNT');
insert into students (first_name, last_name, major) values 
   ('Bill','Gates','ICS');
   
insert into sections (prefix, number, credits) values 
   ('CSNT','110',3);
insert into sections (prefix, number, credits) values 
   ('CSNT','280',3);
insert into sections (prefix, number, credits) values 
   ('ICS','111',4);
   
insert into signups (student_id, section_id) values (1,1);
insert into signups (student_id, section_id) values (1,3);
insert into signups (student_id, section_id) values (2,2);
insert into signups (student_id, section_id) values (2,3);
insert into signups (student_id, section_id) values (3,2);

After running this SQL script, here is a select query that makes use of the view: 


select * from signups_view;
st_id  | first_name | last_name | major | sect_id | prefix | number | credits 
-------+------------+-----------+-------+---------+--------+--------+---------
     1 | Jane       | Doe       | CSNT |       1  | CSNT   | 110    |       3
     1 | Jane       | Doe       | CSNT |       3  | ICS    | 111    |       4
     2 | John       | Doe       | CSNT |       2  | CSNT   | 280    |       3
     2 | John       | Doe       | CSNT |       3  | ICS    | 111    |       4
     3 | Bill       | Gates     | ICS  |       2  | CSNT   | 280    |       3

As can be seen, Jane Doe signed up for CSNT110 and ICS111. John Doe signed up for CSNT280 and ICS111, and Bill Gates signed up for CSNT280.

This view represents unnormalized data that comes from multiple tables. That is the purpose of a view. In other words, a view is used to gather data from more than one table. While views are often the most useful way of representing data to an end user, the data in a view is not normalized and can represent difficulty if you are moving the data into a new database. If that is the situation, you need to have a way of normalizing the data before inserting into the new database tables. That is a topic that we will discuss later.
