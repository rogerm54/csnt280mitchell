Objective 2 - Using SQL scripts to create relational database tables

Let's go over another example of normalizing and creating a relational database. For this example, assume that we want to make a simple database that keeps track of students signing up for sections of courses. Note that since a student can sign up for several sections and each section can have multiple students, the relationship between students and sections is a many-to-may relationship. We need to normalize this data to make a proper relational database.

Database design

Identifying the entities

The first step is to identify the important entities. One entity is a student. Another entity is a section for a course. A third entity is some kind of record of student signups for a section. So, let's create three tables: students , sections and signups .

Adding fields to the tables

Let's add fields to these tables. To keep things simple, we will choose just enough fields to illustrate the concepts. A real student registration system would keep track of a lot more fields than what we will use for our example. Let's start with the students table. For each student, let's keep track of their first name, last name and major. Here is a simple representation of a student:

Although this is very simple, we have followed one of the important rule that fields should be atomic . In this case, we created separate fields for the first name and last name, instead of having one field for the entire name. Let's write the SQL script needed to create this table: 


create table students(
   id serial,
   first_name text,
   last_name text,
   major text,
   primary key (id)
);

Although we could have fit this SQL on fewer lines, it is a good practice to use one line per field. Line 1 is the start of the SQL create statement. Since we are creating a table, we use: 


create table <tablename>

Line 2 declares a field called id that will use the data type serial . This means that id will be a non-null integer that will automatically increment based on a sequence called 'students_id_seq'. This makes the id field a suitable choice for a primary key field as is designated on line 6. A primary key field must be unique for the table. That means that no two records can have the same value for a primary key field. A primary key field is the field that is used to relate (link) database tables together. That is why it is so important for the primary key field to be unique, because specifying the value of a primary key uniquely identifies the record for that table.

Lines 3, 4 and 5 define the first_name, last_name and major fields. These are all given the data type of 'text'. Although we could use a data type of varchar , using a data type of text is more flexible (any size) and makes things simpler (without any performance costs).

Next, we can design the sections table. Here is the SQL script to do this: 


create table sections(
   id serial,
   prefix text,
   number text,
   credits integer,
   primary key (id)
);

For the sections table we separate out the prefix from the number. So, a section like CSNT110, has the prefix CSNT stored in a separate field from the number 110 . The number field is stored as text instead of a number, because there are some courses like ICS110P, where the number also contains a letter. The credits field is declared as an integer. It is possible for some courses to have a number of credits that is not an integer, but for this example we will only use an integer number of credits. Just as with the students table, the id field for the sections table will serve as the primary key.

Note that this is definitely an incomplete set of fields. If there were two sections of CSNT110 or CSNT280, this table definition would not work. In a real database, we would add other fields like location, meeting times, instructor, and section number to allow for more than one section of a course. But, for this example this definition is good enough.

Finally, let's design the signups table. Here is the SQL script to do this: 


create table signups(
   id serial,
   student_id integer references students(id),
   section_id integer references sections(id),
   primary key (id)
);

Line 3 and line 4 are the important lines that link the other two tables. The relationships defined on these lines are what make this a relational database. Line 3 declares a field called student_id with a data type of integer. The additional specifiers, 


references students(id)

will create a foreign key constraint . This foreign key constraint will only allow a record to be stored in the signups table if the value for the student_id exists in the students table. This makes sense as there is no point to storing a student id for a nonexistent student.

Line 4 declares a field section_id with a data type of integer. Just as with the student_id field, the additional specifiers, 


references sections(id)

will create a foreign key constraint that requires the value for a section id exists in the sections table.

Creating a SQL script to create the tables

Let's create the SQL script that creates these tables. The SQL script will be written in a way that would allow for each changes to the design of the database tables. Here is the SQL script: 


-- init.sql
drop table if exists students cascade;
drop table if exists sections cascade;
drop table if exists signups cascade;

create table students(
   id serial,
   first_name text,
   last_name text,
   major text,
   primary key (id)
);
create table sections(
   id serial,
   prefix text,
   number text,
   credits integer,
   primary key (id)
);
create table signups(
   id serial,
   student_id integer references students(id),
   section_id integer references sections(id),
   primary key (id)
);

Lines 2-4 drop the tables if they exist. This will allow us to make changes to the tables and rerun 'init.sql' to change the database design. When you are developing a database, it is very important to be able to make changes at an early stage (before inserting any real data). The use of a SQL script like the one above makes this easy to do.

If you actually create a database and run 'init.sql', the tables would be created. If we connected to the database using psql , we could use the display command to look at the signups table:
