Objective 2 - Using SQL INSERT to populate a table

Inserting Rows of data

Once you have created a database table, you perform an insert query to insert a row of data into that table. The basic syntax for an insert query is as follows: 


insert into <tablename> (field1, field2, field3) values (arg1, arg2, arg3);

That is, you specify the table name, a list of fields and then the values to insert into those fields. When you are inserting literal strings for a value, you must enclose the literal string with single-quotes. You cannot use double-quotes to enclose a literal string.

A simple example

Consider a simple database for a school library. For this database, you want to keep track of users (people who borrow books), books and transactions (which user borrowed which book). A possible "init.sql" to create the tables for this database would be: 


-- init.sql
drop table if exists users cascade;
drop table if exists books cascade;
drop table if exists transactions cascade;

create table users (
   id serial,
   first_name varchar(30),
   last_name varchar(40),
   e_mail varchar(100),
   password varchar(40),
   primary key (id)
);

create table books (
   id serial,
   title text,
   author text,
   category text,
   isbn text,
   primary key (id)
);

create table transactions (
   id serial,
   user_id integer references users(id),
   book_id integer references books(id),
   borrow_date date,
   primary key (id)
);

Let's insert some data into these tables with a SQL script. Here is a script "add_initial_data.sql" that can be used to do this: 


-- add_initial_data.sql
insert into users (first_name,last_name,e_mail,password)
   values ('Jane','Doe','janedoe@gmail.com', md5('janey'));
insert into users (first_name,last_name,e_mail,password)
   values ('John','Doe','johndoe@gmail.com', md5('john'));
insert into users (first_name,last_name,e_mail,password)
   values ('Bill','Gates','billg@hotmail.com', md5('windows'));

insert into books (title,author,category,isbn) values
   ('The Cat in the Hat','Dr. Seuss','childrens','111111');
insert into books (title,author,category,isbn) values
   ('Prey','Michael Crichton','sci-fi','111112');
insert into books (title,author,category,isbn) values
   ('Ender''s Shadow','Orson Scott Card','sci-fi','111113');

insert into transactions (user_id,book_id,borrow_date) values
   (1,1,'10-20-2001');
insert into transactions (user_id,book_id,borrow_date) values
   (2,1,'11-25-2001');
insert into transactions (user_id,book_id,borrow_date) values
   (3,2,'11-20-2002');
insert into transactions (user_id,book_id,borrow_date) values
   (3,3,'11-20-2002');
insert into transactions (user_id,book_id,borrow_date) values
   (1,2,'02-25-2003');

By including the "init.sq" script again, the tables are cleared out before the data is inserted using the "add_initial_data.sql" script.

Let's examine the "add_initial_data.sql" script in more detail. Lines 2-7 insert rows into the users table. Note that you should never store passwords in plain text, so the md5() function is used to hash the passwords. Unlike encryption, hashing is a one way process. A hashed password cannot be converted back into the original password. Instead, you hash the password entered by the user to see if that hash matches the stored hash.

Lines 9-14 insert rows into the books table. These are just regular insert queries, but note on line 14 how an apostrophe is added to a literal string. That is, we want to say the title of the book is "Ender's Shadow". To place an apostrophe inside a literal string, you must use a special escape character. You use a double apostrophe like this:

'Ender''s Shadow'

So, using two single quotes in a row inside a literal string (which must be single-quoted) is how to include an apostrophe.

Lines 16-24 insert rows into the transactions table. Lines 16-17 are inserting a record that Jane Doe borrowed The Cat in the Hat on October 20, 2001. Note how PostgreSQL converts the literal string '10-20-2001' into a valid date. Lines 18-19 insert a record that John Doe borrowed The Cat in the Hat on November 25, 2001, and so forth.

If you run a select query on the transactions table, you will see this: 


sql_example_db=> select * from transactions;
id | user_id | book_id | borrow_date 
----+---------+---------+-------------
  1 |       1 |       1 | 2001-10-20
  2 |       2 |       1 | 2001-11-25
  3 |       3 |       2 | 2002-11-20
  4 |       3 |       3 | 2002-11-20
  5 |       1 |       2 | 2003-02-25

Note how the date is displayed in the default format for PostgreSQL and that the order is different from the literal string that was used to insert the record. Also, note that just displaying this table is not really informative as you cannot figure out who borrowed what book without looking at the users and books tables. To obtain a more informative look, you would want to create a view for the database. Creating views is covered on this page .

One more note about the users table. A better definition would put a constraint on the e_mail field. The constraint that should be used for that field is the unique constraint. Here is the revised version of "init.sql" that does this: 


-- init.sql
drop table if exists users cascade;
drop table if exists books cascade;
drop table if exists transactions cascade;

create table users (
   id serial,
   first_name varchar(30),
   last_name varchar(40),
   e_mail varchar(100) unique,
   password varchar(40),
   primary key (id)
);

create table books (
   id serial,
   title text,
   author text,
   category text,
   isbn text,
   primary key (id)
);

create table transactions (
   id serial,
   user_id integer references users(id),
   book_id integer references books(id),
   borrow_date date,
   primary key (id)
);

Note on line 10 that the unique constraint has been added. This will make it so that no two records in the users table can have the same e_mail value. That is an important constraint to use, as this will be important if when the users table is used for authentication purposes.
